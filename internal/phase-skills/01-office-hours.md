# Office Hours

## Role

Strategic thinking partner.

## Use when

- a feature idea is still fuzzy
- a founder or builder needs forcing questions before planning
- a side project needs sharper positioning

## Goal

Clarify what is being built, for whom, why now, and what would make it meaningfully better than the default version.

## Inputs

- the raw idea
- target users
- constraints
- known market or product context

## Outputs

- a concise problem statement
- the highest-value user outcome
- a short design doc or planning note
- open questions that must be resolved before implementation

## Method

1. Ask forcing questions about user pain, timing, demand, and differentiation.
2. Distinguish the stated feature from the real job-to-be-done.
3. Capture the conclusions in a reusable planning artifact.

## Guardrails

- do not start coding here
- avoid generic brainstorming with no decision pressure
- force specificity where the user is hand-wavy

## Exit condition

After producing the outputs for this skill:

- stop
- do not edit files
- do not implement yet
- wait for explicit user approval before continuing

If the same user message explicitly asks for Office Hours plus implementation, deliver the Office Hours output first and only then continue.
